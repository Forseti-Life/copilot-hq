## Problem
Running `composer install` in `/home/ubuntu/forseti.life/sites/dungeoncrawler` succeeds, but `./vendor/bin/drush status` throws:
```
PHP Fatal error: Uncaught TypeError: Drush\Boot\BootstrapManager::bootstrap():
Return value must be of type Drush\Boot\DrupalBoot8, Drush\Boot\EmptyBoot returned
```

## Root cause
Drush 13 uses `InstalledVersions::getInstallPath('drupal/core')` (via `DrushDrupalFinder::getDrupalRoot()`) to locate the Drupal web root. After a fresh `composer install` in the forseti path, `vendor/composer/installed.php` maps `drupal/core` to `__DIR__ . '/../drupal/core'` (the Composer-managed stub in `vendor/drupal/core/`), NOT to `web/core/` where the full Drupal source lives.

Because the stub directory fails `validRoot()` (missing `autoload.php`, `core/includes/common.inc`, etc.), drush falls back to `EmptyBoot`, which cannot be cast to `DrupalBoot8`.

Additionally, the PSR-4 maps in `autoload_psr4.php` and `autoload_static.php` map `Drupal\Core\` to `$vendorDir/drupal/core/lib/Drupal/Core` (stub â€” empty) instead of `$baseDir/web/core/lib/Drupal/Core`.

## Fix (applied)
Three files patched manually after `composer install`:

1. **`vendor/composer/installed.php`** â€” change drupal/core `install_path`:
   ```php
   // Before:
   'install_path' => __DIR__ . '/../drupal/core',
   // After:
   'install_path' => __DIR__ . '/../../web/core',
   ```

2. **`vendor/composer/autoload_psr4.php`** â€” fix `Drupal\Core\` and `Drupal\Component\` paths:
   ```php
   // Before:
   'Drupal\\Core\\' => array($vendorDir . '/drupal/core/lib/Drupal/Core'),
   'Drupal\\Component\\' => array($vendorDir . '/drupal/core/lib/Drupal/Component'),
   // After:
   'Drupal\\Core\\' => array($baseDir . '/web/core/lib/Drupal/Core'),
   'Drupal\\Component\\' => array($baseDir . '/web/core/lib/Drupal/Component'),
   ```

3. **`vendor/composer/autoload_static.php`** â€” replace all `__DIR__ . '/..' . '/drupal/core/` occurrences with `__DIR__ . '/../../web/core/` (28â†’3 remaining after sed, the 3 remaining are for `core-composer-scaffold` etc, not drupal/core itself).

4. **`web/autoload.php`** â€” created (scaffold file not generated by composer install):
   ```php
   return require __DIR__ . '/../vendor/autoload.php';
   ```

5. **`drush/drush.yml`** â€” created to set stable root/uri defaults:
   ```yaml
   options:
     root: /var/www/html/dungeoncrawler/web
     uri: https://dungeoncrawler.forseti.life
   ```

## Notes
- The `/home/ubuntu/forseti.life/sites/dungeoncrawler/` path uses `dungeoncrawler_dev` database (not production `dungeoncrawler`).
- The production path `/var/www/html/dungeoncrawler/` already works correctly without these fixes.
- These vendor file patches will be overwritten by a future `composer install`. If regeneration is needed, re-apply the three edits above or create a composer post-install script.
- The root cause is the `drupal-core` Composer package type: the installer normally puts it into `web/core/` but the installed.php records it at the vendor default location when the script is generated before the installer runs.
